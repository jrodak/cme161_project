Network Packet Visualization

CME 161: Data Visualization Spring 2016

Jordan Rodak and Alberto Huerta

While browsing the web, we rarely think about the complex systems and technologies that make it all possible. We simply type a URL into our browsers and then, like magic, we receive the requested page. This project will allow us to dive into the often ignored details of our web browsing. Using Wireshark we captured all network packets received and sent by a laptop on a university WiFi network. We captured packets in the following scenarios for 15 seconds each:

  1. Idle, no web browser open
  2. Browsing Google and search results
  3. Browsing NYTimes homepage and articles
Total Number of Packets Captured

There are two primary points to take away from this data. First, we can see that even while idle our computers are sending and receiving packets. In this case these packets were most DNS messages. Second, we can see that sites such as the New York Times which display a large number of images and advertisements than sites like Google, where most most of the content is simple text. Click on each of the bars to filter the data by experiment.

Number of Packets Captured over time

Here we see what's commonly referred to as the "burstiness" of network traffic. Instead of a relatively continous stream of packets, we see that the packets are sent and receivied in bursts in a short amount of time after a web request is made.

Packet Statistics

The structure of packets sent over the Internet are specified by various standardized protocols that all server different purposes. These protocols are also layered, meaning a packet of one protocol often contains a packet of another protocol. For example, when our browser sends an HTTP request to a website, the HTTP request is wrapped in a TCP packet, which is then subsequently wrapped by an IP packet and an Ethernet packet. We'll learn more about these protocols and their purposes below.
